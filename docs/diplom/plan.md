## Введение

- Зачем сделан Chronos.
- Сравнение с профессиональными инструментами (Kubernetes, Portainer) и их ограничения для сценария автора.
- Вдохновение Azure: управление конфигурацией приложений из кода; желание реализовать аналог на базе Docker Compose.
- Обоснование выбора Docker / контейнеризации.

## Теоретическая часть

1. **Три проекта:** `Chronos.Core` (работа с контейнерами из кода), `Chronos.Agent` (конвейер развёртывания и управления на узле), `Chronos.Master` (ядро кластера: агенты, маршрутизация, координация).
2. **Общая архитектура:** публикация, стратегии бэкапирования томов, маршрутизация на Master, роль агентов и наблюдение за контейнерами; тесты и jobs (упомянуть).
3. **Core:** устройство, решаемые задачи, публикация; механизм восстановления Fluent-кода из `docker-compose.yml` (`ComposeYamlParser`, `FromYaml`, round-trip); выгрузка томов; артефакты деплоя и тесты как артефакты.
4. **Agent:** работа, мониторинг контейнеров, копирование/снимки томов; взаимодействие с Master.
5. **Master:** учёт агентов, маршрутизация трафика к агентам, HAProxy.
6. **Наблюдаемость:** Loki и Grafana.
